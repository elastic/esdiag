## Why

"Acquisition" is overloaded: today a diagnostic can arrive either by ESDiag pulling
live product APIs (`Collect`) or by ESDiag reading a bundle the product itself
generated (`Load`), but nothing states which products each path applies to. This lets
`Collect` be (mis)read as a general "get any diagnostic" verb, conflates a deliberate
scope boundary (we will never API-collect Elastic Agent or a platform) with unfinished
work (Kibana/Agent *processing* is in progress), and leaves "why did this skip?"
ambiguous. Rationale: **ADR-0019** (with the `Skipped` refinement in **ADR-0016**).

## What Changes

- Define the **Collect** stage as scoped to the **API-collectable** products only —
  **Elasticsearch, Kibana, Logstash**. `Collect` SHALL NOT target Elastic Agent or any
  platform (ECE, ECK, KubernetesPlatform).
- Establish that **product-provided** diagnostics — Elastic Agent (generates its own
  bundle) and platform diagnostics (ECE/ECK/KubernetesPlatform, generated by the
  platform) — enter the pipeline **exclusively via `Load`**. A job over them is
  `Load → [Process] → …`, never `Collect`.
- Codify the two distinct gap kinds so they are never conflated:
  **out-of-scope-by-design** (Agent + platform-level API collection) vs
  **not-yet-implemented** (Kibana processing on a branch; Agent processing, PR293).
  Both surface as a skip today but mean opposite things (ties to the `Skipped`
  subtype in ADR-0016).
- Note (not implement) a future **"trigger generation then Load"** delegated
  acquisition flow, distinct from API `Collect` — see design.
- **Not breaking:** no persisted shape, manifest field, or CLI flag changes; this
  formalizes an existing boundary. It narrows what a `Collect` request may legitimately
  target, which surfaces as a clearer refusal where a caller previously got an
  undifferentiated unsupported-product error.

## Capabilities

### New Capabilities

- _(none — this modifies existing capabilities)_

### Modified Capabilities

- `collection-execution`: add that the `Collect` stage is scoped to the API-collectable
  products (Elasticsearch, Kibana, Logstash) and refuses Agent/platform targets with
  by-design guidance to use `Load`.
- `included-diagnostic-jobs`: add that product-provided (Agent + platform) diagnostics
  are acquired via `Load`, never `Collect`; a platform bundle is Loaded then fanned out;
  ECE carries no application data (no included diagnostics); and a recognized-but-
  unprocessable child is the *not-yet-implemented* gap, distinct from the by-design
  Collect-scope gap.

## Impact

- **Core:** the `Receiver` resolution for the input phase — `Collect` receivers are
  constructed only for Elasticsearch/Kibana/Logstash; Agent/platform inputs resolve to
  a `Load` receiver. The collection definition / registry (ADR-0005) only ever
  describes API sources for the three API-collectable products.
- **CLI:** `esdiag collect` refuses an Agent/platform target with a by-design message
  pointing to `read`/`Load`; `esdiag read` (`Load`) remains the entry for those.
- **Web UI:** the `Collect` panel's remote-collect option applies to the API-collectable
  products; product-provided bundles arrive through `Upload` (`Load`).
- **Rejected suggestions:** `cli-agent-mode` (unrelated — it governs low-noise CLI for
  *LLM* agent invocations, not the Elastic Agent product) and
  `kibana-diagnostic-collection` (Kibana is in-scope; ADR-0019 affirms, does not change,
  its collection contract).
